# Полная документационная схема разработки rldyourterm v1.0.0

Дата: 2026-02-24. Версия: `1.0.0` (целевой релизный baseline).

## 1) Назначение документа
Это операционный каркас всех документов v1.0.0. Он определяет:
- что мы делаем в v1.0,
- какие требования неизменяемы,
- в каком порядке закрываем этапы,
- какие артефакты считаются обязательными,
- какие проверки проходят вручную перед релизом.

## 2) Утверждённые факты v1.0

1. Ядро проекта: один терминал (single-window) как baseline.
2. Приоритеты v1.0: стабильность > AI-совместимость > скорость.
3. Технический подход: Rust + VSA-слои + внешние адаптеры (нет заимствованного terminal-core).
4. Режимы рендера: `cpu`, `gpu`, `auto`.
5. Shell baseline: `fish + Starship`, fallback на `zsh`; Bash — не целевой по умолчанию для v1.0 на Linux/macOS.
6. Визуальный стиль: стартовый профиль `cuberpunk`.
7. Основной UX: in-terminal command palette (`Ctrl/Cmd + Shift + P`), не только конфиг-файлы.
8. Обновления и релизы: без CI, только ручные проверки и ручной релиз-процесс.
9. Целевая ОС: Ubuntu 22.04 LTS, 24.04 LTS, 25.10, macOS; Windows-платформа заложена архитектурно, не в активном v1.0 scope.
10. Стабильность как сервисное требование: GPU падение ≠ падение сессии.

## 4) Step-by-step roadmap для v1.0 (документация → код)

### Этап 0. Lock требований
- Финализировать ответы по `planning/discovery/v1.0.0-questionnaire` в `answer-lock`.
- Проверить, что все ADR согласованы и `Accepted`.
- Убедиться, что все non-goals зафиксированы в `AGENTS.md`.

### Этап 1. Архитектурный фундамент
- Подтвердить слойные границы и адаптерную модель.
- Зафиксировать внутренние API между `core`, `services`, `features`, `ui`, `app`.
- Подготовить ADR по shell и fallback strategy.

### Этап 2. Каркас интеграций
- Зафиксировать интеграционные границы для `portable-pty`, `winit`, `wgpu`.
- Зафиксировать правила `retry`, `backoff`, `degrade`, `controlled stop`.
- Подключить наблюдаемость (`event-id`, `tracing`, telemetry events).

### Этап 3. MVP runtime
- Реализовать базовый single-window терминал.
- Поддержать shell bootstrap и fallback.
- Реализовать `cpu`, `gpu`, `auto` и ручной switch.
- Встроить command palette для runtime-настроек.

### Этап 4. Стабилизация и hardening
- Добавить bounded recovery при GPU и PTY ошибках.
- Реализовать resize-throttle и coalescing.
- Зафиксировать поведение для long-session/idle/burst сценариев.
- Выполнить ручной стресс-календарь.

### Этап 5. release lock
- Сверить `AGENTS.md`, ADR, roadmap, quality gates, риски.
- Обновить все `metrics/version/1.0.0/*.md` до `1.0.0`.
- Заполнить `planning/operations/v1.0.0-manual-test-plan.md` и закрыть `release-pack`.

## 5) Acceptance критерия перехода этапов
Этап закрывается только если:
- все связанные ADR имеют статус `Accepted`;
- требования и ответы в `answer-lock` не конфликтуют с `quality-gates`;
- ключевые риски имеют mitigation и owner;
- есть готовый manual-test evidence до перехода к следующему этапу;
- версия `1.0.0` закреплена во всех целевых метриках и документах.

## 6) Принцип контроля
- Любая новая техническая интеграция сначала описывается в `planning/stack/*`.
- Любой новый риск/ограничение добавляется в `planning/risk/v1.0.0-risk-matrix.md`.
- Если документы расходятся, приоритет: AGENTS -> ADR -> roadmap -> quality -> stack -> operations.
